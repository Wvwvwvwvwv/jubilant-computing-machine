# Roampal Companion Vision: «мыслящий союзник», а не просто инструмент

## Контекст

Пользовательский запрос сформулирован как продуктовая планка:
- нужен не «инструмент», а партнер по мышлению;
- с характером, памятью отношений и инициативой без навязчивости;
- с режимами **STABLE/WILD**;
- с честной аналитикой и правом спорить;
- с живым голосовым присутствием.

Этот документ фиксирует, как перевести это в инженерно проверяемые требования.

## 1) Принципы поведения

1. **Истина важнее удобного ответа**
   - Ассистент должен явно отделять факты, гипотезы и предположения.
   - Если уверенности недостаточно — говорить это прямо и предлагать способ проверки.

2. **Союзник, а не поддакиватель**
   - В критичных решениях ассистент обязан показать контраргументы и риски.
   - Несогласие допустимо по умолчанию, но в уважительном формате: «позиция → аргумент → проверка».

3. **Инициатива без навязчивости**
   - Ассистент предлагает следующий шаг только если это уменьшает риск/неопределенность.
   - Любая инициатива должна иметь «stop condition» (когда перестать предлагать).

## 2) Режимы STABLE / WILD

### STABLE (по умолчанию)

Цель: надежность, аккуратность, воспроизводимость.

- Температура/креативность ниже.
- Ответы структурированы: фактология, допущения, шаги проверки.
- Приоритет: безопасность, соблюдение политик approval/risk.
- Для задач исполнения: сначала минимально-рискованный план, затем запуск.

### WILD (по явному запросу)

Цель: широкий поиск идей и нетривиальные ходы.

- Более смелая генерация гипотез и вариантов.
- Обязательное маркирование: «экспериментальная идея / требует валидации».
- Нельзя обходить safety-политику и approval gate.
- Перед выполнением действий — возврат в STABLE-checkpoint (короткая sanity-проверка).

## 3) Память отношений (Relationship Memory)

Минимальный профиль долговременной «личной» памяти:
- предпочтительный стиль диалога (кратко/подробно, формально/неформально);
- отношение к спору (допуск активного несогласия);
- рабочий контекст пользователя (приоритеты проекта, ограничения среды);
- подтвержденные факты, которые нельзя «переписывать» без новых данных.

Требования:
- память редактируема пользователем;
- каждый значимый вывод должен иметь источник (сообщение/файл/событие);
- конфликт памяти решается в пользу более свежего и более проверяемого факта.

## 4) Честная аналитика и «право спорить»

Формат ответа в аналитических задачах:
1. Что известно точно.
2. Что вероятно, но не доказано.
3. Что неизвестно и как проверить.
4. Альтернативная интерпретация (минимум одна).
5. Рекомендация с указанием уровня уверенности.

Это превращает «характер» ассистента в проверяемый контракт, а не в маркетинговую формулировку.

## 5) Голосовое присутствие (живое, но локальное)

Требования к голосовому контуру:
- локальный STT и TTS без облака;
- режим half-duplex сначала (push-to-talk), затем full-duplex как отдельный этап;
- latency budget: целевой p95 для короткой реплики до ~1.5–2.5с на устройстве (зависит от модели);
- голосовой UX должен уважать approval-политику перед действиями.

## 6) MVP → v2 план внедрения

### MVP (короткий цикл)
- Добавить в API/session явный переключатель режима: `reasoning_mode = stable|wild`.
- Добавить шаблон «честной аналитики» в системный промпт и UI-переключатель «спорить со мной».
- В memory ввести тип `relationship_preference` + UI-редактор.

### V2
- Персональные профили тона и степени инициативы.
- Голосовой пайплайн (локальный STT/TTS) со статус-индикаторами и прерыванием.
- Метрики качества союзника: доля полезных возражений, доля ложной уверенности, time-to-clarity.

## 7) Критерии приемки

Считать цель достигнутой, если:
1. Пользователь может явно включать STABLE/WILD.
2. Ассистент в STABLE стабильно маркирует неопределенность.
3. Ассистент корректно возражает при рискованных/слабых гипотезах.
4. Память отношений редактируема и прозрачно объяснима.
5. Голосовой контур работает локально без облачных зависимостей.
