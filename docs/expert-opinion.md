# Квалифицированное (скептическое) мнение по проекту Roampal Android

Дата: 2026-03-13
Ветка: work

## Важная оговорка

В предоставленном контексте нет самих inline-комментариев ревью (только сводка diff). Поэтому ниже — проверяемая экспертная оценка по фактическому состоянию кода и запуску тестов.

## Сильные стороны (подтверждено)

1. **Архитектурная цель «локально-first» действительно соблюдается**:
   - Core API, Embeddings, Frontend и KoboldCpp разделены на сервисы.
   - Есть отдельные Termux-скрипты под setup/deploy/smoke/diagnose.

2. **Tasks стек стал заметно зрелее**:
   - Planner: LLM-first с детерминированным fallback.
   - TaskRunner: SQLite persistence, события, approval fingerprint/policy, retry и классификация ошибок.
   - В run-эндпоинте есть защита от drift: задача перечитывается из БД перед фактическим запуском.

3. **Операционная надежность улучшена**:
   - В deploy есть rollback через `last_good` + preflight schema check.
   - Есть integrity-скрипт проверки равенства локального HEAD и origin/work.

4. **Тестовое покрытие есть и не декоративное**:
   - Покрыты planner, task runner, API задач, сериализация chat, Termux-детекция sandbox.

## Что вызывает осторожность (не критика ради критики)

1. **Sandbox hardening в реальном Android-окружении зависит от best-effort веток**.
   Это нормально для Termux, но важно не переоценивать гарантии seccomp/rlimits как «жестко enforced» во всех устройствах.

2. **PDF/OCR часть функциональна, но зависит от внешних бинарников/пакетов**.
   Без корректной установки OCR-пайплайн может отработать частично (или только text-layer).

3. **Сложность setup/deploy растет**.
   Добавленные fallback-ветки правильные, но их нужно постоянно подтверждать device-level smoke-логами на целевых Android/Termux окружениях.

4. **CI проверяет backend в Linux-контуре, но не эмулирует Termux-специфику**.
   Это стандартно, но важно помнить: «зелёный CI» ≠ «гарантированно зелёный девайс».

## Итоговое заключение

Проект уже перешел из «демо-песочницы» к **практичному локальному ассистенту с инженерными предохранителями**.

Если оценивать строго:
- **Концептуально:** очень хорошая траектория.
- **Технически:** заметно сильнее среднего pet-проекта.
- **Операционно:** ключевой remaining-risk — именно device variability (Termux/Python/OCR/system libs).

## Что я бы считал «готовностью к стабильной эксплуатации»

Минимальный объективный чеклист:
1. Несколько подряд успешных запусков `termux/deploy.sh work` и `termux/full-smoke.sh` на целевом устройстве.
2. Подтверждение OCR-пайплайна на реальном скан-PDF (не только text-layer PDF).
3. Регулярная проверка parity (`HEAD == origin/work`) и integrity script после каждого фикса.
4. Периодический пересмотр fallback-логики под обновления Python/Termux.
